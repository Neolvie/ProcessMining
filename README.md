# Process Mining — Directum RX

Система анализа эффективности бизнес-процессов на основе логов `WorkflowProcessService` Directum RX.
Автоматически разбирает логи, строит аналитику и находит узкие места без ручной настройки.

```bash
docker compose up -d
# → http://localhost:8001
```

---

## Что это такое

**Process Mining** — дисциплина анализа данных, которая восстанавливает фактическое поведение процессов
из журналов событий (event log), а не из проектной документации.

Ключевые понятия:

| Термин | Значение в системе |
|--------|--------------------|
| **Экземпляр (case/trace)** | Одно выполнение бизнес-процесса: от старта до завершения или прерывания |
| **Событие (event)** | Строка лога: активация блока, завершение, прерывание, сбой |
| **Блок (activity)** | Конкретный шаг схемы процесса (задача, согласование, уведомление и т.п.) |
| **Схема (scheme_id)** | Конкретная версия шаблона маршрута в Directum RX |
| **Переход (directly-follows)** | Факт того, что блок B следует за блоком A в том же экземпляре |
| **Прерывание (abort)** | Экземпляр или блок завершился досрочно без штатного результата |
| **Длительность** | Время от активации блока/процесса до его завершения |

## Быстрый старт

### Требования
- Docker и Docker Compose
- Логи `WorkflowProcessService` в формате JSON-Lines

### Структура папок

```
ProcessMining/
├── logs/              # ← кладите сюда лог-файлы (.log или .log.*)
│   ├── host1_2025-01-01.log
│   ├── host1_2025-01-02.log
│   └── ...
├── metadata/          # ← JSON-файлы метаданных сущностей Directum RX (опционально)
├── .env               # ← настройки AI (скопируйте из .env.example)
└── docker-compose.yml
```

### Запуск

```bash
# 1. Скопируйте логи
cp /path/to/your/logs/*.log ./logs/

# 2. (опционально) настройте AI
cp .env.example .env
# отредактируйте .env — вставьте OPENROUTER_API_KEY

# 3. Запуск
docker compose up -d

# 4. Откройте браузер
# http://localhost:8001
```

### Остановка

```bash
docker compose down
```

## Формат логов

Система читает **JSON-Lines** файлы из папки `logs/`. Каждая строка — JSON-объект:

```json
{"@t":"2025-01-15T06:23:41.123Z","@mt":"Block activated in process","InstanceId":"abc-123","ProcessId":"guid","SchemeId":1049,"BlockId":"ApprovalBlock","Host":"server-01"}
```

Поддерживаемые события (`@mt`):

| Шаблон | Тип события |
|--------|------------|
| `Block activated / started in process` | Активация шага |
| `Block completed in process` | Завершение шага |
| `Block aborted in process` | Прерывание шага |
| `Process (instance) started` | Старт экземпляра |
| `Process (instance) completed` | Завершение экземпляра |
| `Process (instance) aborted` | Прерывание экземпляра |
| `Lock contention detected` | Конкуренция за объект |
| `Failed to process message` / `Error processing` | Сбой обработки |

**Несколько хостов** поддерживаются автоматически — дубликаты сообщений RabbitMQ фильтруются
по `(instance_id, block_id, event_type, timestamp)`.

## Загрузка данных за длительный период

### Как добавить логи за месяц и более

Просто положите все файлы в папку `logs/` и перезапустите:

```bash
# Архив за январь–февраль
cp /archives/2025-01/*.log ./logs/
cp /archives/2025-02/*.log ./logs/

docker compose up -d --force-recreate
```

Система разберёт все файлы при старте — это занимает от нескольких секунд до пары минут в зависимости от объёма.

### Ориентиры по производительности

| Период | Строк лога | Событий | Разбор | DuckDB |
|--------|-----------|---------|--------|--------|
| 4 дня | ~330 000 | ~94 000 | ~5 с | ~0.5 с |
| 1 месяц | ~2.5 млн | ~700 000 | ~35 с | ~3 с |
| 3 месяца | ~7.5 млн | ~2 млн | ~100 с | ~8 с |

*Приблизительно для контейнера 2 CPU / 4 GB RAM. Пока приложение загружает данные, браузер показывает статус «Загрузка данных…» и автоматически обновляется.*

### Что изменится в интерфейсе при большом периоде

- **Временная шкала** — переключайтесь в режим **«По дням»**, чтобы видеть тренды за весь период (а не 700+ часовых столбцов)
- **Тепловая карта** — покажет паттерны нагрузки по дням недели и часам суток за весь период
- **Гистограмма длительностей** — при накоплении данных видны выбросы и изменения поведения
- **Дашборд** → период в заголовке отражает реальный диапазон данных
- **Топ процессов** — при большом периоде выбросы одного дня не искажают статистику

### Советы для работы с большим объёмом

1. На страницах **Блоки** и **Потоки** применяйте фильтр по процессу — без него грузятся топ-200 блоков
2. На **Временной шкале** для периодов > 2 недель используйте детализацию **«По дням»**
3. Актуальные JSON в `metadata/` ускоряют распознавание имён процессов

## Страницы интерфейса

### Дашборд
Главная сводка: KPI-карточки, активность по часам, тепловая карта, распределение длительностей, топ процессов.

**Цветные чипы под KPI** — автоматические выводы:
- 🔴 Красный — критическая проблема (завершаемость < 70%, много прерываний у лидера)
- 🟡 Жёлтый — предупреждение (длинный P95, много блокировок)
- 🟢 Зелёный — критических проблем не обнаружено

**Тепловая карта активности** — матрица «день недели × час суток». Тёмно-оранжевые клетки — пиковые часы нагрузки. Помогает понять: в какое время процессы стартуют чаще всего, есть ли ночная активность, как нагрузка распределяется по рабочей неделе.

**Гистограмма длительностей** — распределение завершённых экземпляров по времени выполнения. Нормальный процесс даёт одну вершину. Два пика или длинный хвост → часть экземпляров идёт по нетипичному, долгому пути.

### Временная шкала
Показывает **динамику во времени**: сколько процессов стартовало, завершилось и прервалось в каждый час (или день).

Используйте для обнаружения:
- Пиков нагрузки (rush hours, конец месяца)
- Аномальных провалов активности (потенциальные простои)
- Дней с высоким числом прерываний

Второй график — **активации блоков** (общее количество шагов) и **среднее время обработки сообщений** (мс).
Рост `avg_span_ms` при пиковой нагрузке → признак перегрузки очереди RabbitMQ или медленного ответа БД.

### Процессы
Таблица всех типов процессов с ключевыми метриками. Кликните на строку → откроется детальный модал:
- Активность конкретного типа по часам
- Статистика по блокам (avg, P95)
- Последние 50 экземпляров

### Блоки / Этапы
Аналитика по отдельным шагам (блокам) маршрутов. Самые медленные блоки — первые кандидаты на оптимизацию.

`P95 >> среднего` → есть редкие, но очень долгие случаи выполнения (например, ожидание внешнего ответа или зависший исполнитель).

### Потоки переходов (DFG)
**Directly-Follows Graph** — какой блок за каким следует и как часто. Это основа Process Mining:
- Частые переходы — «главная дорога» процесса
- Редкие переходы — исключения и нетипичные ветки
- Большой `avg_gap_sec` между соседними блоками → ожидание пользователя или внешней системы

### Узкие места
Три вкладки:
- **Медленные блоки** — наибольшее среднее/P95 время выполнения
- **Медленные процессы** — типы процессов с долгим средним временем
- **Высокий % прерываний** — блоки, которые чаще других не доходят до конца

### Проблемы и аномалии
- **Прерывания по процессам** — какие типы процессов чаще всего не завершаются
- **Блокировки** — конкуренция за объекты (lock contention) по часам; пики → узкое место в базе данных
- **Долгие экземпляры** — зависшие или патологически долгие кейсы
- **Сбои** — ошибки обработки сообщений (Failed spans)

### AI Анализ
Генерирует структурированный анализ на русском языке: ключевые проблемы, рекомендации по оптимизации, приоритетные действия.
Требует `OPENROUTER_API_KEY` в `.env`.

```bash
docker compose up -d --force-recreate
```

## Идеи для развития

На основе лучших практик Process Mining (Celonis, ProM, VK Process Mining):

### Фильтрация без перезапуска
- **Фильтр по периоду** — выбор диапазона дат прямо в UI (передаётся как query-параметр к API)
- **Фильтр по инициатору** — cl/js/it-старты: ручные vs автоматические процессы
- **Фильтр по хосту** — изолировать нагрузку одного сервера
- **Фильтр по результату блока** — только успешные / только прерванные пути

### Анализ вариантов (Variant Analysis)
- Группировать экземпляры по **уникальной последовательности блоков** (variant)
- Показывать топ-N вариантов с долей от общего числа
- **Happy path** — самый частый вариант как эталонный маршрут; всё остальное — отклонения
- Процент экземпляров, следующих happy path — ключевой KPI стабильности процесса

### Визуальная карта процесса
- DFG как **интерактивный граф** (узлы = блоки, рёбра = переходы, толщина = частота)
- Наложение метрики: цвет узла = средняя длительность (зелёный → красный)
- Кликнуть на блок → список экземпляров, прошедших через него

### Анализ рекурсии и доработок
- Блоки, через которые экземпляр проходит **больше одного раза** — пересогласования, возвраты на доработку
- Процент экземпляров с рекурсией = KPI качества (чем выше, тем больше итераций на обработку)

### SLA-мониторинг
- Настраиваемые пороги: блок > N часов → нарушение SLA
- Дашборд нарушений: сколько экземпляров нарушили SLA за период
- Тренд нарушений по дням

### Сравнение периодов
- Сравнение текущей недели с предыдущей (дельта KPI)
- Выявление регрессий: «стало хуже» / «стало лучше»

## Архитектура

```
logs/ ──parse──▶ events[] ──build──▶ DuckDB (in-memory)
                                          │
                          ┌───────────────┼───────────────┐
                          ▼               ▼               ▼
                     FastAPI REST    Static SPA      SSE stream
                     /api/*          index.html      /api/ai-insights
                          │               │               │
                          └───────────────┴───────────────┘
                                          │
                                    Browser (JS)
```

**Стек:**
- Backend: Python 3.11, FastAPI, DuckDB, pandas, orjson
- Frontend: Vanilla JS (16 модулей), Chart.js 4, Directum UI kit
- AI: OpenAI-compatible API (OpenRouter), SSE streaming
- Deploy: Docker, single container
